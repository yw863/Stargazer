# StarGazer — Impeccable Design Context

## Design Context

### Users

Urban astronomy enthusiasts in China (Yangtze River Delta), beginner-to-intermediate level. Comfortable with technical terminology (Bortle scale, apparent magnitude, altitude angle, elongation). They come to the tool with a specific intent — "Is tonight worth going?" — and need efficient, confident answers. They are not here to discover, browse, or socialise. They are operating an instrument.

### Brand Personality

**Three words:** precise · quiet · expectant

The experience should feel like opening a well-made field notebook — not a consumer app, not a public information service. There is anticipation in the interface, but it is restrained: the quiet before a clear night, not the excitement of a social feed. Data is the hero. The interface serves data.

### Reference & Anti-Reference

**Reference — Are.na aesthetic:**
- Whitespace used for breathing room, not filler
- Data placed directly on screen without card containers wrapping everything
- Color used to convey information, not decoration
- Linear / line-based decorative elements: elegant, intellectual, spare

**Explicit anti-references (never look like these):**
- SkySafari and similar apps: realistic star photography, glow effects, "space blue" gradients, neon-on-dark
- NASA public outreach pages: large hero banners, exclamation headlines, full-bleed photography
- Social / community products: like counts, comment threads, avatar rows, engagement visual language
- SaaS dashboards: white backgrounds, rounded card stacks, blue CTA buttons, dashboard-grid layouts

### Aesthetic Direction

**Primary motif — Ephemeris table aesthetic:**  
Tabular star catalogue and almanac aesthetics. Monospace numerals in fixed-width columns, light hairline dividers, data aligned as if typeset for a printed ephemeris. Numbers are not data labels — they are the visual texture of the page. The interface should feel closer to a JPL Horizons output or a printed tide table than to a modern web app.

**Theme:** Deep dark. Background near-black with a blue-navy tint (`#080c14`). This is a product used at night, outdoors or in a dark room.

**Color philosophy:** Color communicates state and hierarchy, never decoration. Accent (`#5b8dd9`, low-saturation star blue) is used sparingly — it signals "this is the thing that matters right now." No gradients as decoration. No glow effects.

**Copy style (non-negotiable):**
- Middle-dot `·` separated readout phrases, not sentences
- No exclamation marks, no emoji, no colloquial phrasing
- Astronomy terminology stays professional — no dumbing down
- Locked copy: view one subtitle "Do Look Up——", view one heading "近日可见："

### Typography System

Three-layer font system — each layer has a precise role:

| Layer | Usage | Font |
|-------|-------|------|
| **Display** | Headings, event names, section titles | Spectral (or Cormorant — user's explicit choice; elegant serif with well-optimised screen rendering) |
| **Body / UI labels** | Descriptions, form labels, secondary text | DM Sans (lightweight, unobtrusive — does not compete with display or numerics) |
| **Numeric readouts** | Magnitude, distance, angular separation, time, all pure-number values | Geist Mono (monospace, number alignment, precision instrument quality) |
| **Chinese text** | All Chinese UI copy and data labels | Noto Serif SC（思源宋体 Google Fonts 版，作为 Spectral / DM Sans 的中文 fallback） |

Avoid Inter and other pan-use grotesques. Numeric values must always render in Geist Mono — this is not optional.

### Design Principles

1. **Data is the visual.** Numbers and values are the design material, not text labels. Lay them out as a typesetter would set an almanac.

2. **No card for everything.** Cards imply containment and separation. Use whitespace, typographic hierarchy, and hairline rules instead. Reserve visible containers for genuinely elevated surfaces.

3. **One accent, used once.** The accent color (`#5b8dd9`) marks the single most important thing on screen. When everything is accented, nothing is.

4. **Instrument, not publication.** Information density should be high enough to feel like a readout, not so sparse it feels like a landing page. The user is operating the tool, not reading about it.

5. **Motion as state change, never spectacle.** Transitions confirm that something happened. They should be swift (150–250ms), ease-out, opacity + transform only. Support `prefers-reduced-motion` — reduce all motion to instant cuts.

### Accessibility

- Contrast: WCAG AA minimum throughout
- Motion: `prefers-reduced-motion` supported — all animations reduce to instant state change
- No pure red / pure green as sole carriers of critical information (use shape, label, or additional cue)
- No hard requirement for dedicated color-blind mode
