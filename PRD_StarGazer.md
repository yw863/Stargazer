## 1. 产品概述

### 1.1 一句话描述

一个帮助无车城市天文爱好者快速找到「去哪里、哪天去、怎么去」的观测规划工具。

### 1.2 背景与痛点

城市天文爱好者在规划一次观测时，需要交叉查询 5-6 个不同信息源：天文事件参数（亮度、方位、时间窗口）、候选地点的光污染等级、天气与云量预测、公共交通路线与耗时。这些信息分散在不同平台，且需要人工综合判断，决策成本高、耗时长。尤其对于无法自驾的观测者，交通方式的筛选进一步增加了复杂度。

### 1.3 目标用户

身处城市、依赖公共交通（或有限自驾能力）的天文入门至中级爱好者。地理范围 MVP 阶段聚焦长三角地区周边。

### 1.4 核心价值主张

将分散在多个平台的观测决策信息自动聚合，输出一份「开箱即用」的观测出行方案。用户不再需要逐一查阅光污染地图、天气预报、地图导航和天文资讯，一次输入即可获得综合推荐。

## 2. 产品基调与文案规范

### 2.1 设计基调

整体基调：**神秘、简约、深邃、科学、优雅**。

补充关键词：**克制的浪漫**——不是社交媒体式的"绝美星空！"兴奋感，而是天文观测者独有的安静期待感；**精密工具感**——它是帮助决策的仪器，不是内容社区或科普平台；**优雅、人文、格调**——不是冷硬刻板的“科幻感”页面，而是有互动感、有呼吸感，古典星图的优雅浪漫与精密观测的简约克制相得益彰。

视觉方向（供前端 Agent 参考，不做强制约束）：深色系为主，呼应夜空；排版适当留白，信息密度适中；数据呈现优先于装饰元素。

### 2.2 语言风格

- **中文为主**，天文术语使用专业表达（Bortle 等级、星等、高度角、方位角），不做过度通俗化——目标用户是天文爱好者，他们理解这些概念。
- **仪表盘读数式**：优先使用结构化短语，用中点「·」分隔，而非完整叙述句。像星图上的标注，不像导游解说词。
- **语气克制、陈述性**：呈现事实与数据，不做感叹、不加语气词。

### 2.3 关键文案方向

以下为各视图核心位置的文案风格示例，具体措辞待定，供前端开发参照调性：

**视图一 · 事件总览**

- 页面副标题：「Do Look Up——」（致敬科幻电影 *Don't Look Up*，传递"抬头看，值得看"的态度，为天文爱好者提供一个会心一笑）。
- 副标题下方换行，主标题：「近日可见：」。
- 主标题下方直接展示事件卡片。
- 事件卡片状态描述采用读数式，例如：「目视星等 ~5.1 · 肉眼极限附近 · 昏星」而非「现在很亮！肉眼就能看到」。
- 即将上线的事件标记文案：「即将上线」（简洁、不解释）。

**视图二 · 观测规划输入**

- 不设引导语或欢迎语，直接展示输入表单。交互本身即引导。
- 提交按钮文案建议：「生成观测方案」或「规划观测」。

**视图三 · 推荐结果**

- 观测条件摘要采用读数式排列，例如：「推荐日期 4月22日 · 预测星等 4.8 · 昏星 · 残月 22:30落 · 观测窗口 20:30–22:15」。
- 避免使用「最适合」「最佳」等主观评价词，用「条件最优」或直接以数据呈现。
- 若用户所选日期范围内存在不可观测日（太阳角距过小），以提示信息告知：「4月25日 · 太阳角距 3.7° · 不可观测」。

### 2.4 产品名称

暂定「StarGazer」，中英文名待定。命名方向供参考：偏工具感（如「观星指北」）或偏意境感（如「暗域」）。最终名称确定后更新全文档。

## 3. 用户流程

```
用户打开网站
    → 视图一：浏览近期天文事件总览
    → 点击某个事件（视图原地过渡，卡片展开为视图二，顶部保留事件标识与返回入口）
    → 视图二：输入观测约束条件（所在城市、所在地经纬度、日期范围、交通方式、设备）
    → 提交
    → 系统依次执行：天象分析 → 地点筛选 → 环境评估 + 交通规划（并行） → 综合排序
    → 视图三：展示推荐结果（地图 + 对比表格 + 观测方案卡片）
```

**视图转场说明**：视图一→视图二不使用分栏布局或页面跳转，采用原地展开过渡。用户点击事件卡片后，页面平滑过渡为视图二的输入表单，顶部保留一行事件标识（如「C/2025 R3 彗星」+ 返回箭头），让用户保持上下文感知。

## 4. 功能需求

### 4.1 视图一：天文事件总览

**目的**：回答用户「最近有什么可以看的」。

**展示元素（每个事件为一张卡片）**：

- 事件名称（如「C/2025 R3 彗星」）
- 事件类型标签（彗星 / 流星雨）
- 最佳观测时间范围（如「2026年4月中旬 - 5月上旬」）
- 一句话当前状态描述，采用读数式（如「目视星等 ~5.1 · 肉眼极限附近 · 昏星」）
- 交互状态标识：可点击进入 / 即将上线（灰色不可点击）

**MVP 内容**：

| 事件 | 类型 | 交互状态 |
|------|------|---------|
| C/2025 R3 彗星 | 彗星 | 可点击，完整功能 |
| 英仙座流星雨（2026年8月） | 流星雨 | 占位，标记「即将上线」 |
| 双子座流星雨（2026年12月） | 流星雨 | 占位，标记「即将上线」 |

**数据来源**：手动维护的事件配置文件。彗星的「当前状态描述」由天象分析 Agent 动态生成（基于 COBS 最新数据 + Horizons 太阳角距与晨昏星判断）。

---

### 4.2 视图二：观测规划输入

**目的**：收集用户的约束条件，作为后端 Agent 的输入参数。

**输入字段**：

| 字段 | 类型 | 是否必填 | 说明 |
|------|------|---------|------|
| 所在位置 | 浏览器定位 / 手动输入 | 必填 | 通过浏览器 Geolocation API 自动获取经纬度，同时反查显示城市名称供用户确认或手动修改。手动输入时自动匹配经纬度。 |
| 计划观测日期 | 日期范围选择器 | 必填 | 最少 1 天，最多 3 天连续范围 |
| 交通方式 | 单选 | 必填 | 选项：「可自驾」/「无车」 |
| 观测设备 | 单选 | 选填 | 选项：「肉眼」/「双筒望远镜」/「望远镜」，默认值「肉眼」 |

**交互说明**：

- 日期选择器应限制可选范围为当天起未来 14 天内（超出 14 天的天气预测无参考价值）。
- 提交后进入加载状态，展示进度提示（因后端涉及多个 API 调用，预计耗时 10-30 秒）。
- 用户位置输入后，前端自动将其转换为经纬度坐标（调用高德地理编码 API），后续所有 Agent 均直接使用经纬度作为输入。

### 4.3 视图三：推荐结果

**目的**：展示综合推荐方案，帮助用户做出最终决策。

#### 4.3.1 观测条件摘要

展示所选日期范围中**综合条件最优的一天**的概览信息，采用读数式排列：

- 推荐日期（如「推荐日期 4月22日（周三）」）
- 目标天体预测亮度（如「预测星等 4.8」）
- 晨昏星标识（如「昏星 · 日落后西方低空可见」或「晨星 · 日出前东方低空可见」）
- 太阳角距（如「太阳角距 13.8°」）
- 月相与月光影响（如「残月 · 22:30 落 · 影响低」），月亮照明度与角距数据来自 Horizons
- 最佳观测时间窗口（如「观测窗口 20:30–22:15」）
- 目标方位与高度角概述（如「方位 西北偏北 330° · 高度角 15°–25°」）

**不可观测提示**：若用户所选日期范围内某天太阳角距 < 15°，该日期标记为不可观测，摘要中注明原因。若所选范围内全部日期均不可观测，直接提示用户并建议调整日期。

#### 4.3.2 地图

- 显示用户当前位置与所有候选观测地点
- 地点标记按综合推荐等级以颜色区分
- 点击标记可查看地点简要信息

#### 4.3.3 对比表格

横向对比所有候选地点，支持按列排序。

| 列名 | 数据来源 | 说明 |
|------|---------|------|
| 地点名称 | 知识库 | |
| Bortle 等级 | 知识库 | 1-9，数值越低越暗 |
| 交通方式 | 高德地图 API | 如「地铁+公交」「自驾」 |
| 交通时长 | 高德地图 API | 单程预估时间（公共交通时长参与排序） |
| 云量预测 | 7Timer | 按所选日期范围中条件最优日展示 |
| 透明度评级 | 7Timer（仅 ASTRO 有效时） | 大气透明度（1-8，越低越好） |
| 综合评分 | 编排层计算 | 加权评分，满分 100 |

如用户选择了多天日期范围：表格默认展示条件最优日的数据，提供切换控件可查看其他日期。

#### 4.3.4 观测方案卡片

用户点击表格中某一地点后展开的详情信息：

- 交通路线摘要（起点→终点，途经方式，预估时长）
- 若用户选择「可自驾」，同时展示公共交通和自驾两种方案
- 当晚目标天体的方位与高度角随时间变化信息（来自 Horizons topocentric 查询）
- 建议的观测时间窗口
- 注意事项（如「该地点东侧有城镇光害，当晚目标方位偏东，可能受影响」）

              │
         ┌────┴────┐
         ▼         ▼
    Agent 3:    Agent 4:

    （筛选+    （不筛选，

         │         │

**各 Agent 职责边界**：

| Agent | 是否筛选 | 是否评分 | 调用外部 API |
|-------|---------|---------|-------------|
| Agent 1 天象分析 | 否（仅判断可观测性） | 否 | Horizons（缓存 + 实时 topocentric） |
| Agent 2 地点筛选 | **是**（距离 + 交通方式 + Bortle，三层过滤） | 否 | 无（纯计算 + RAG 检索） |
| Agent 3 环境评估 | **是**（天气一票否决） | **是**（天气评分） | 7Timer |
| Agent 4 交通规划 | 否 | 否（但交通时长参与排序） | 高德地图 |
| 编排层 | 否 | **是**（综合评分 + 排序） | 无 |

### 5.2 Horizons 双查询架构

Agent 1 对 JPL Horizons API 的调用分为两种模式，职责不同：

**查询 A：每日定时拟合查询（地心坐标，后台定时任务）**

- 目的：获取 r（日心距离）和 Δ（地心距离）用于彗星亮度预测模型的拟合与更新。
- 执行频率：每日凌晨自动执行一次。
- CENTER：`500@399`（地心）
- QUANTITIES：`'9,19,20,23,25'`
- STEP_SIZE：`'1d'`
- 时间范围：历史起始日（COBS 最早记录）至未来 30 天
- 输出字段及用途：

| Horizons 字段 | 含义 | 用途 |
|--------------|------|------|
| r (Quantity 19) | 日心距离（AU） | 光变模型拟合与预测 |
| delta (Quantity 20) | 地心距离（AU） | 光变模型拟合与预测 |
| T-mag (Quantity 9) | Horizons 内置亮度预测 | 与自建模型对比验证 |
| S-O-T (Quantity 23) | 太阳角距（°）及 /L 或 /T | 判断可观测性 + 晨昏星标识 |
| T-O-M / MN_Illu% (Quantity 25) | 月亮角距与照明度 | 月光干扰评估 |

- 处理流程：拉取 COBS 最新观测 → 提取每日中位表观星等 m → 匹配 Horizons 对应日期的 r 和 Δ → 回归拟合 H 和 n → 缓存拟合结果及未来 30 天的 r、Δ、S-O-T、月亮数据

**查询 B：用户实时观测规划查询（topocentric 坐标，用户触发）**

- 目的：获取目标天体在用户所在地的方位角（AZ）和高度角（EL）随时间变化的数据。
- 执行时机：用户提交观测规划请求时实时调用。
- CENTER：`'coord'`，配合 SITE_COORD 设置为用户城市的经纬度（如 `'121.47,31.23,0'` 上海）
- COORD_TYPE：`'GEODETIC'`
- QUANTITIES：`'4'`（Apparent AZ & EL）
- STEP_SIZE：`'30m'`（每 30 分钟一个数据点）
- 时间范围：用户所选日期范围内每天的日落前 1 小时至日出后 1 小时
- 输出：目标天体在当晚的方位角、高度角随时间变化序列，用于确定最佳观测窗口和朝向

**两种查询的关系**：查询 A 提供"值不值得去看"（亮度预测、是否可观测、月光干扰），查询 B 提供"怎么看"（朝哪个方向、什么时间段、仰角多高）。查询 A 的结果可缓存复用，查询 B 必须按用户位置实时调用。

### 5.3 7Timer 天气数据降级策略

7Timer 的 ASTRO 和 CIVIL 产品预报窗口不同，且 ASTRO 产品存在更新延迟的已知问题。Agent 3 采用以下降级策略：

**优先级 1：ASTRO 产品（72 小时预报）**

- 请求 URL：`https://www.7timer.info/bin/astro.php?lon={lon}&lat={lat}&ac=0&unit=metric&output=json&tzshift=0`
- 可用字段：cloudcover、transparency、prec_type、temp2m
- 适用条件：用户所选日期在 ASTRO 预报覆盖范围内，**且** init 时间距当前不超过 24 小时（即数据未过期）

**优先级 2：CIVIL 产品（7 天预报，降级方案）**

- 请求 URL：`https://www.7timer.info/bin/civil.php?lon={lon}&lat={lat}&ac=0&unit=metric&output=json&tzshift=0`
- 可用字段：cloudcover、prec_type、temp2m（**无 transparency**）
- 适用条件：用户所选日期超出 ASTRO 72 小时窗口，或 ASTRO 数据过期
- 影响：天气评分仅基于云量，不含透明度。对比表格中透明度列显示「—」

**优先级 3：无天气数据**

- 适用条件：用户所选日期超出 CIVIL 7 天窗口
- 影响：该日期天气评分标注「天气预测暂不可用」，综合排序暂按光污染和交通排列。建议用户临近日期再查。

**7Timer timepoint 解读**：`init` 字段为预报初始化时间（如 `2026041206` = 2026年4月12日06:00 UTC），`timepoint` 为相对 init 的小时偏移。实际时间 = init + timepoint 小时。ASTRO 的 timepoint 范围为 3 到 72（每 3 小时一个点），CIVIL 的 timepoint 范围为 3 到 168。

### 5.4 各 Agent 详细定义

#### Agent 1：天象分析 Agent

**职责**：根据天文事件和观测日期，输出观测条件需求。不做地点筛选。

**输入**：

```json
{
  "event_id": "C2025R3",
  "date_range": ["2026-04-22", "2026-04-24"],
  "equipment": "binoculars",
  "user_location": {"latitude": 31.23, "longitude": 121.47}
}
```

**处理逻辑（以彗星为例）**：

1. 读取每日定时任务缓存的拟合结果（H、n），以及用户所选日期范围内的 r、Δ。
2. 代入 H 和 n，计算用户所选日期的预测星等。
3. 读取缓存的太阳角距（S-O-T）数据。若用户所选日期的 S-O-T < 15°，标记该日期为「不可观测」，附注原因。
4. 读取缓存的 /L 或 /T 标识，判断目标为晨星或昏星。
5. 读取缓存的月亮角距与照明度（T-O-M / MN_Illu%），评估月光干扰等级。
6. 调用 Horizons 查询 B（topocentric），获取用户位置的 AZ/EL 随时间变化数据，确定观测窗口和方位。
7. 根据预测星等和用户设备，计算建议的最低 Bortle 等级。

**亮度 → Bortle 映射规则**：

| 预测星等 | 肉眼建议 Bortle | 双筒建议 Bortle | 望远镜建议 Bortle |
|---------|----------------|----------------|-----------------|
| ≤ 3.0 | ≤ 7 | ≤ 8 | ≤ 9 |
| 3.1 - 4.5 | ≤ 5 | ≤ 7 | ≤ 8 |
| 4.6 - 5.5 | ≤ 4 | ≤ 5 | ≤ 7 |
| 5.6 - 6.5 | ≤ 3 | ≤ 4 | ≤ 5 |
| ≥ 6.6 | ≤ 2 | ≤ 3 | ≤ 4 |

**太阳角距判断规则**：

| S-O-T | 判定 |
|-------|------|
| < 15° | 不可观测，该日期直接排除 |
| 15° - 25° | 可观测但条件受限，观测窗口极短，需提示用户 |
| > 25° | 正常可观测 |

**晨昏星判断**：Horizons 输出的 S-O-T 字段末尾标记 `/L`（Leading）表示晨星（日出前可见），`/T`（Trailing）表示昏星（日落后可见）。此标识直接影响用户的出行时间规划。

**输出「观测需求单」**：

```json
{
  "event_id": "C2025R3",
  "dates": [
    {
      "date": "2026-04-22",
      "observable": true,
      "predicted_magnitude": 4.6,
      "horizons_magnitude": 7.0,
      "max_bortle": 4,
      "solar_elongation": 13.8,
      "visibility_type": "evening",
      "best_window": {"start": "19:30", "end": "21:00"},
      "azimuth_at_peak": 290,
      "altitude_at_peak": 18,
      "direction_description": "西偏北",
      "moon_illumination_percent": 27.0,
      "moon_target_angle": 73.7,
      "moon_impact": "低"
    },
    {
      "date": "2026-04-25",
      "observable": false,
      "reason": "太阳角距 3.7°，目标过于接近太阳"
    }
  ]
}
```

#### Agent 2：地点筛选 Agent（RAG，唯一筛选环节）

**职责**：从知识库中检索并过滤候选观测地点。所有地点筛选逻辑集中在此 Agent。

**输入**：用户经纬度 + 交通方式 + Agent 1 的观测需求单（含 max_bortle）。

**处理逻辑（三层过滤）**：

1. **距离过滤**：用 Haversine 公式计算用户经纬度与知识库中每个地点的直线距离，排除 > 600km 的地点。此步为纯数学计算，不调用任何外部 API。
2. **交通方式过滤**：若用户选择「无车」，根据知识库中预标注的 `access_modes` 字段，排除仅标记为「自驾」的地点。
3. **Bortle 过滤**：Bortle 等级 ≤ 需求单中的 max_bortle。

按 Bortle 等级升序排列，返回候选列表。

**输出**：

```json
{
  "candidates": [
    {
      "name": "天荒坪观星营地",
      "latitude": 30.58,
      "longitude": 119.68,
      "bortle": 3,
      "access_modes": ["自驾", "公共交通"],
      "notes": "东侧有安吉县城光害，建议面朝西侧观测"
    }
  ]
}
```

#### Agent 3：环境评估 Agent（筛选 + 评分）

**职责**：查询候选地点在观测日期的天气条件，执行天气一票否决，并计算天气评分。

**输入**：候选地点列表（含经纬度）+ 日期范围。

**处理逻辑**：

1. 按 5.3 节降级策略选择数据源（ASTRO → CIVIL → 无数据）。
2. 提取用户所选日期范围内夜间时段（18:00-06:00）的数据点，观测窗口内（Agent 1 的 best_window）时段权重更高。
3. 执行一票否决（见下方规则），被否决的地点-日期组合移除。若某地点在所有可观测日期均被否决，整个地点移除。
4. 对存活的地点-日期组合计算天气评分。

**7Timer Transparency 字段说明（仅 ASTRO 产品）**：

单位为 mag/airmass（每气团消光值），**数值越小天空越通透**：

| 等级 | 消光值 | 含义 |
|------|--------|------|
| 1 | <0.3 | 极佳 |
| 2 | 0.3-0.4 | 优秀 |
| 3 | 0.4-0.5 | 良好 |
| 4 | 0.5-0.6 | 中等 |
| 5 | 0.6-0.7 | 一般 |
| 6 | 0.7-0.85 | 较差 |
| 7 | 0.85-1.0 | 差 |
| 8 | >1.0 | 极差（严重雾霾/水汽） |

**一票否决规则**：

| 条件 | 数据源 | 说明 |
|------|--------|------|
| cloudcover ≥ 8 | ASTRO 或 CIVIL | 云量约 90% 以上 |
| transparency ≥ 7 | 仅 ASTRO | 消光 ≥ 0.85 mag/airmass，严重雾霾 |
| prec_type ≠ "none" | ASTRO 或 CIVIL | 有任何形式降水 |

**注意**：当降级使用 CIVIL 时，无 transparency 数据，仅依据 cloudcover 和 prec_type 判断。

**输出**：

```json
{
  "location": "天荒坪观星营地",
  "weather_source": "astro",
  "weather_by_date": [
    {
      "date": "2026-04-22",
      "cloudcover": 2,
      "transparency": 3,
      "prec_type": "none",
      "temp2m": 14,
      "weather_score": 88
    }
  ]
}
```

当降级使用 CIVIL 时，transparency 字段为 null。

#### Agent 4：交通规划 Agent（不筛选，仅提供数据）

**职责**：查询用户出发地到每个候选地点的交通方案。不做任何筛选或排除。

**输入**：用户出发地经纬度 + Agent 3 存活的候选地点列表 + 交通方式偏好。

**API 调用**：高德地图路径规划 API（免费额度 5000 次/日，足够 MVP 使用）。

**输出逻辑**：
- 若用户选择「无车」→ 仅查询公共交通路线
- 若用户选择「可自驾」→ 同时查询公共交通和自驾两种路线

**输出**：

```json
{
  "location": "天荒坪观星营地",
  "transit_options": [
    {
      "mode": "公共交通",
      "duration_minutes": 180,
      "summary": "地铁2号线 → 高铁至安吉 → 公交 → 步行25分钟",
      "transfers": 3
    },
    {
      "mode": "自驾",
      "duration_minutes": 135,
      "summary": "G50沪渝高速 → S14杭长高速 → 县道"
    }
  ]
}
```

#### 编排层：综合排序

**职责**：汇总 Agent 1-4 的输出，计算综合评分并排序。

**评分权重（MVP）**：

| 维度 | 权重 | 说明 |
|------|------|------|
| 天气评分 | 50% | Agent 3 输出的 weather_score |
| 光污染匹配度 | 25% | Bortle 越低越好，相对于 max_bortle 的余量 |
| 交通便利性 | 15% | 公共交通时长越短越好（公交时长用于排序，最终由用户自行斟酌） |
| 额外加分项 | 10% | 地点已知注意事项与当晚观测方位无冲突 |

**推荐数量上限**：排序后最多保留 **6 个**推荐地点。若经筛选后候选不足 6 个，则全部列出。

**最终输出**：结构化 JSON，包含观测条件摘要 + 排序后的候选地点列表（最多 6 个，各附天气、交通、评分），供前端直接渲染。

## 6. 数据 Schema

### 6.1 观测地点知识库

每条记录包含以下字段：

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| id | string | 是 | 唯一标识 |
| name | string | 是 | 地点名称 |
| region | string | 是 | 所属城市/区域（如「上海周边」「杭州周边」） |
| latitude | float | 是 | 纬度 |
| longitude | float | 是 | 经度 |
| bortle | int (1-9) | 是 | Bortle 光污染等级 |
| access_modes | string[] | 是 | 可达交通方式，如 ["自驾", "公共交通"]。若仅含 "自驾"，则「无车」用户会被 Agent 2 过滤掉 |
| access_notes | string | 否 | 交通补充说明（如「末班公交 21:00」） |
| terrain_notes | string | 否 | 地形说明（如「东侧有山脊遮挡」） |
| facilities | string | 否 | 设施说明（如「有停车场」「无厕所」） |
| source | string | 否 | 信息来源（如「小红书用户推荐」「实地验证」） |

### 6.2 天文事件配置

手动维护的事件元数据：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| event_id | string | 如 "C2025R3" |
| name | string | 如 "C/2025 R3 彗星" |
| type | enum | "comet" / "meteor_shower" |
| status | enum | "active" / "coming_soon" |
| best_period | string | 如 "2026年4月中旬 - 5月上旬" |
| cobs_designation | string | COBS API 参数，如 "C/2025 R3" |
| horizons_id | string | JPL Horizons 查询 ID，如 "C/2025 R3" |

### 6.3 每日拟合缓存

由每日定时任务生成并缓存：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| event_id | string | 事件标识 |
| fit_date | date | 拟合执行日期 |
| H | float | 拟合绝对星等 |
| n | float | 拟合活动参数 |
| ephemeris | array | 未来 30 天逐日数据：r、Δ、S-O-T、/L或/T、T-O-M、MN_Illu%、Horizons T-mag |

### 10.1 MVP 包含

- C/2025 R3 彗星的完整规划流程

## 12. 已关闭的决策记录

| 编号 | 问题 | 决策 | 依据 |
|------|------|------|------|
| 1 | 观测地点知识库地理范围 | 长三角地区 | 便于面试时实机演示 |
| 2 | 地图 API 选型 | 高德地图 | 用户和地点均在中国大陆，免费 5000 次/日足够 MVP |
| 3 | 天气 API 选型 | 7Timer（ASTRO 优先 + CIVIL 降级） | ASTRO 专为天文设计含 transparency；ASTRO 更新不稳定时降级至 CIVIL 用 cloudcover |
| 4 | 彗星亮度预测模型更新频率 | 每日一次定时拟合，缓存结果 | 彗星亮度日间变化小，每日更新足够；避免用户查询时实时拟合的延迟 |
| 5 | Horizons 查询架构 | 双查询（地心每日定时 + topocentric 用户实时） | AZ/EL 必须基于用户位置，与拟合用的 r/Δ 查询职责不同 |
| 6 | 太阳角距安全阈值 | < 15° 标记为不可观测 | C/2025 R3 在近日点前后角距低至 3.7°，必须拦截 |
| 7 | 交通一票否决规则 | 取消，改为 600km 距离预筛 | 距离计算无需调用 API，更快更省；交通时长仅用于排序，由用户自行斟酌 |
| 8 | Agent 职责分工 | Agent 2 为唯一筛选环节；Agent 4 不筛选只提供数据 | 筛选逻辑集中化，职责边界更清晰 |
| 9 | 超出 72h 天气数据 | 降级至 CIVIL（7天），超出 7 天标注不可用 | 有 cloudcover 总比没有好，但不给不完整的评分误导用户 |
| 10 | Transparency 方向 | 数值越小越好（1=极佳，8=极差），≥ 7 一票否决 | 单位为 mag/airmass 消光值，来源 7Timer 官方 Wiki |
| 11 | Horizons 响应解析方式 | 项目文件中的独立预处理脚本（`backend/tools/horizons_parser.py`），不放在 Dify Agent 节点中 | 纯文本解析不涉及 LLM 推理，独立脚本可被每日定时任务和用户实时查询复用，且更易单独测试调试 |
| 12 | 用户位置经纬度转换 | 前端处理（调用高德地理编码 API） | 减少后端复杂度，前端已接入高德 JS API，顺便复用 |
